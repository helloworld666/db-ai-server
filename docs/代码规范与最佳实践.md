# db-ai-server 代码规范与最佳实践 (v5.0)

## 架构设计原则

### 核心原则：简单优于复杂

**v5.0架构变更**：
- ❌ 移除ReAct循环和工具调用机制
- ❌ 移除复杂的JSON解析逻辑
- ✅ 采用直接SQL生成模式
- ✅ LLM只返回纯SQL，不返回工具调用JSON

**为什么简化？**
- 2026-04-15出现的问题：ToolMessage大JSON"训练"LLM返回超大JSON，导致多层解析失败
- 解决方案：直接让LLM生成SQL，不通过工具调用

## 为什么需要代码规范？

**真实案例**：2026-04-10 出现的缩进错误导致"数据库操做描述不够具体"的错误：
- SQL检查通过，但由于缩进错误，错误提示代码始终执行
- 这是一个典型的低级错误，但影响严重

## 3. 代码风格指南

### 3.1 缩进

```python
# ✅ 正确：使用4个空格缩进
if condition:
    logger.info("Log message")
    return result

# ❌ 错误：使用制表符
if condition:
	logger.info("Log message")
	return result

# ❌ 错误：缩进不一致（本次问题的根源）
if condition:
    logger.info("Log message")

# 这段代码没有缩进，无论条件如何都会执行
error_msg = "错误信息"
return error_msg
```

### 3.2 行长度

```python
# ✅ 正确：每行不超过120字符
result = await mcp_client.call_tool(
    "execute_sql",
    {"sql": sql}
)

# ❌ 错误：行太长
result = await mcp_client.call_tool("execute_sql", {"sql": sql, "params": params, "timeout": 30})
```

### 3.3 字符串

```python
# ✅ 推荐：使用f-string
logger.info(f"SQL检查: {sql[:50]}...")

# ✅ 多行字符串使用三引号
error_msg = """数据库操做描述不够具体。
请提供更明确的描述，例如：
- '更新货架编号为***的货架'
- '更新用户名为***的用户为可用状态'
"""

# ❌ 避免字符串拼接（可读性差）
error_msg = "数据库操做描述不够具体。\n请提供更明确的描述，例如：\n- '更新货架编号为***的货架'"
```

## 4. 避免常见错误

### 4.1 缩进错误

**错误示例**：
```python
if re.search(pattern, text):
    logger.info("Pattern found")

# 这段代码没有缩进！
error_msg = "Error message"
return error_msg
```

**修复**：
```python
if re.search(pattern, text):
    logger.info("Pattern found")
    # 正确缩进
    error_msg = "Error message"
    return error_msg
```

**预防措施**：
- 使用编辑器的"显示空白字符"功能
- 运行 Black 自动格式化
- 使用 EditorConfig 统一缩进风格

### 4.2 复制粘贴错误

**错误场景**：
从不同文件/编辑器复制代码时，缩进可能丢失。

**预防措施**：
```bash
# 粘贴后立即格式化
black src/mcp_server.py
```

### 4.3 混用空格和制表符

**错误示例**：
```python
# 前两行用空格，第三行用制表符
if condition:
    logger.info("Log")
		return value  # 使用了制表符！
```

**预防措施**：
- 配置编辑器将制表符自动转换为空格
- 运行 Black 时会自动统一

## 10. 架构设计规范

### 10.1 禁止过度设计

**❌ 错误示例**（v4.x架构）：
```python
# 过度复杂的ReAct循环
async def _execute_agent_loop(self, messages, query):
    intermediate_steps = []
    for iteration in range(max_iterations):
        # 调用LLM
        response = await self.bound_llm.ainvoke(messages)
        # 检查工具调用
        tool_calls = self._extract_tool_calls(response)
        # 执行工具
        for tool_call in tool_calls:
            result = await self._execute_tool(tool_call)
            messages.append(ToolMessage(content=str(result)))
        # 循环继续...
```

**✅ 正确示例**（v5.0架构）：
```python
# 简单直接的两阶段处理
async def ainvoke(self, query):
    # 步骤1: 获取schema
    schema = self._get_schema()
    # 步骤2: 生成SQL
    sql = await self._generate_sql(query, schema)
    return {"success": True, "sql": sql}
```

### 10.2 LLM初始化规范（LangChain标准）

**必须使用`init_chat_model`**：
```python
from langchain.chat_models import init_chat_model

# 标准方式 - 统一入口，自动识别提供商
llm = init_chat_model("openai:gpt-4o")
llm = init_chat_model("anthropic:claude-3-5-sonnet")
llm = init_chat_model("ollama:llama3.1")
```

**规范要点**：
1. ✅ 使用`init_chat_model`统一入口
2. ✅ 模型标识符格式：`provider:model`
3. ✅ 自动处理不同提供商的初始化差异

**❌ 错误示例**（旧方式）：
```python
# 不要这样做 - 自定义工厂类
class LLMFactory:
    _providers = {...}
    
    def create(self, config):
        provider = config.get("provider")
        provider_class = self._providers.get(provider)
        return provider_class(config).create()
```

### 10.3 工具定义规范（LangChain标准）

**必须使用`@tool`装饰器**：
```python
from langchain.tools import tool
from pydantic import BaseModel, Field

# 1. 定义Pydantic输入模型
class GetSchemaInput(BaseModel):
    """获取数据库Schema的输入参数"""
    table_name: Optional[str] = Field(
        default=None,
        description="可选，指定表名。不提供则返回所有表"
    )

# 2. 使用@tool装饰器定义工具
@tool(args_schema=GetSchemaInput)
def get_database_schema(table_name: Optional[str] = None) -> str:
    """获取数据库Schema信息。

    返回数据库表结构信息，包括表名、字段、类型、注释等。
    如果不提供table_name，返回所有表的信息。

    Args:
        table_name: 可选，指定表名

    Returns:
        JSON格式的Schema信息
    """
    # 实现代码...
```

**规范要点**：
1. ✅ 使用`@tool`装饰器（不要用`StructuredTool.from_function`）
2. ✅ 函数docstring自动成为工具描述
3. ✅ 使用Pydantic模型定义`args_schema`
4. ✅ 用`Field(description=...)`描述参数
5. ✅ 在docstring中用`Args:`说明参数

**❌ 错误示例**（旧方式）：
```python
from langchain_core.tools import StructuredTool

# 不要这样做
def my_tool():
    pass

tools.append(StructuredTool.from_function(
    func=my_tool,
    name="my_tool",
    description="工具描述",  # 单独传入
    args_schema=MyInput
))
```

### 10.4 LLM交互规范

**❌ 禁止**：
- 让LLM返回工具调用JSON格式
- 在ToolMessage中发送大JSON给LLM
- 要求LLM进行多步推理

**✅ 应该**：
- 让LLM直接返回纯SQL
- 在提示词中提供必要的schema信息
- 一次调用获取结果

### 10.5 响应解析规范

**❌ 禁止复杂解析**：
```python
# 不要这样做 - 过度复杂的解析
def parse_response(self, content):
    # 尝试解析JSON
    # 尝试解析Markdown
    # 尝试解析Python dict
    # 尝试修复截断JSON
    # ...多层嵌套解析
```

**✅ 简单解析**：
```python
# 直接提取SQL
def _extract_sql(self, content):
    # 清理Markdown
    text = re.sub(r'^```.*?\n?', '', content)
    # 提取SQL语句
    match = re.search(r'(SELECT|INSERT|UPDATE|DELETE)\s+.+', text, re.I)
    return match.group(0) if match else None
```
