# db-ai-server 代码规范与最佳实践 (v6.0)

## 架构设计原则

### 核心原则：零硬编码、LLM自主决策

**v6.0架构**（LangChain v1.0规范）：
- ✅ 使用ReAct Agent实现工具链调用
- ✅ **LLM自主决定**调用哪些工具、调用顺序
- ✅ **零硬编码**：所有提示词从配置文件加载
- ✅ **零硬编码**：所有业务逻辑由LLM决策，代码不做判断
- ✅ 支持复杂操作如"更新并返回结果"

**禁止事项**：
- ❌ 禁止在代码中硬编码任何提示词
- ❌ 禁止在代码中硬编码数据库表/字段描述
- ❌ 禁止硬编码工具调用逻辑（如"如果XX则调用YY"）
- ❌ 禁止硬编码业务判断逻辑

## 为什么需要代码规范？

**真实案例**：2026-04-10 出现的缩进错误导致"数据库操做描述不够具体"的错误：
- SQL检查通过，但由于缩进错误，错误提示代码始终执行
- 这是一个典型的低级错误，但影响严重

## 3. 代码风格指南

### 3.1 缩进

```python
# ✅ 正确：使用4个空格缩进
if condition:
    logger.info("Log message")
    return result

# ❌ 错误：使用制表符
if condition:
	logger.info("Log message")
	return result

# ❌ 错误：缩进不一致（本次问题的根源）
if condition:
    logger.info("Log message")

# 这段代码没有缩进，无论条件如何都会执行
error_msg = "错误信息"
return error_msg
```

### 3.2 行长度

```python
# ✅ 正确：每行不超过120字符
result = await mcp_client.call_tool(
    "execute_sql",
    {"sql": sql}
)

# ❌ 错误：行太长
result = await mcp_client.call_tool("execute_sql", {"sql": sql, "params": params, "timeout": 30})
```

### 3.3 字符串

```python
# ✅ 推荐：使用f-string
logger.info(f"SQL检查: {sql[:50]}...")

# ✅ 多行字符串使用三引号
error_msg = """数据库操做描述不够具体。
请提供更明确的描述，例如：
- '更新货架编号为***的货架'
- '更新用户名为***的用户为可用状态'
"""

# ❌ 避免字符串拼接（可读性差）
error_msg = "数据库操做描述不够具体。\n请提供更明确的描述，例如：\n- '更新货架编号为***的货架'"
```

## 4. 避免常见错误

### 4.1 缩进错误

**错误示例**：
```python
if re.search(pattern, text):
    logger.info("Pattern found")

# 这段代码没有缩进！
error_msg = "Error message"
return error_msg
```

**修复**：
```python
if re.search(pattern, text):
    logger.info("Pattern found")
    # 正确缩进
    error_msg = "Error message"
    return error_msg
```

**预防措施**：
- 使用编辑器的"显示空白字符"功能
- 运行 Black 自动格式化
- 使用 EditorConfig 统一缩进风格

### 4.2 复制粘贴错误

**错误场景**：
从不同文件/编辑器复制代码时，缩进可能丢失。

**预防措施**：
```bash
# 粘贴后立即格式化
black src/mcp_server.py
```

### 4.3 混用空格和制表符

**错误示例**：
```python
# 前两行用空格，第三行用制表符
if condition:
    logger.info("Log")
		return value  # 使用了制表符！
```

**预防措施**：
- 配置编辑器将制表符自动转换为空格
- 运行 Black 时会自动统一

## 10. 架构设计规范

### 10.1 禁止过度设计和硬编码

**❌ 错误示例**（硬编码业务逻辑）：
```python
# 禁止：硬编码关键词判断
if any(kw in query for kw in ["返回", "结果", "查看"]):
    # 硬编码的业务逻辑
    select_sql = self._generate_select_for_update(sql)

# 禁止：硬编码工具调用顺序
async def process(self, query):
    # 步骤1: 必须获取schema
    schema = await self.tools["get_database_schema"]()
    # 步骤2: 必须生成SQL
    sql = await self.tools["generate_sql"](query, schema)
    # 步骤3: 必须执行SQL
    result = await self.tools["execute_sql"](sql)
```

**✅ 正确示例**（v6.0架构 - LLM自主决策）：
```python
# Agent只提供工具，LLM决定调用顺序
async def ainvoke(self, query):
    # 绑定工具到LLM
    llm_with_tools = self.llm.bind_tools(self.tools)
    
    # LLM自主决定调用哪些工具
    response = await llm_with_tools.ainvoke(messages)
    
    # 如果LLM请求工具调用，执行它
    if response.tool_calls:
        for tool_call in response.tool_calls:
            result = await self.execute_tool(tool_call)
            messages.append(ToolMessage(content=str(result)))
        # 继续循环，让LLM决定下一步
    
    # 返回LLM的最终结果
    return response.content
```

### 10.3 工具定义规范（LangChain标准）

**必须使用`@tool`装饰器**：
```python
from langchain.tools import tool
from pydantic import BaseModel, Field

# 1. 定义Pydantic输入模型
class GetSchemaInput(BaseModel):
    """获取数据库Schema的输入参数"""
    table_name: Optional[str] = Field(
        default=None,
        description="可选，指定表名。不提供则返回所有表"
    )

# 2. 使用@tool装饰器定义工具
@tool(args_schema=GetSchemaInput)
def get_database_schema(table_name: Optional[str] = None) -> str:
    """获取数据库Schema信息。

    返回数据库表结构信息，包括表名、字段、类型、注释等。
    如果不提供table_name，返回所有表的信息。

    Args:
        table_name: 可选，指定表名

    Returns:
        JSON格式的Schema信息
    """
    # 实现代码...
```

**规范要点**：
1. ✅ 使用`@tool`装饰器（不要用`StructuredTool.from_function`）
2. ✅ 函数docstring自动成为工具描述
3. ✅ 使用Pydantic模型定义`args_schema`
4. ✅ 用`Field(description=...)`描述参数
5. ✅ 在docstring中用`Args:`说明参数

**❌ 错误示例**（旧方式）：
```python
from langchain_core.tools import StructuredTool

# 不要这样做
def my_tool():
    pass

tools.append(StructuredTool.from_function(
    func=my_tool,
    name="my_tool",
    description="工具描述",  # 单独传入
    args_schema=MyInput
))
```

### 10.4 ReAct Agent 工具链规范（v6.0）

**✅ 使用LangChain v1.0 bind_tools实现工具链**：
```python
from langchain_core.tools import tool
from pydantic import BaseModel, Field

# 1. 定义工具（使用@tool装饰器）
@tool
async def get_database_schema(table_name: str = "") -> str:
    """获取数据库Schema信息"""
    # 实现...

@tool
async def execute_sql(sql: str) -> str:
    """执行SQL语句"""
    # 实现...

# 2. 创建Agent（LLM自主决策）
class ReActAgent:
    def __init__(self, llm, tools, system_prompt):
        self.llm = llm.bind_tools(tools)
        self.tools = {t.name: t for t in tools}
        self.system_prompt = system_prompt  # 从配置加载
    
    async def ainvoke(self, query):
        messages = [
            SystemMessage(content=self.system_prompt),
            HumanMessage(content=query)
        ]
        
        while iteration < max_iterations:
            response = await self.llm.ainvoke(messages)
            
            # LLM决定是否调用工具
            if response.tool_calls:
                for tool_call in response.tool_calls:
                    tool_result = await self.tools[tool_call.name].ainvoke(
                        tool_call.args
                    )
                    messages.append(ToolMessage(
                        content=str(tool_result),
                        tool_call_id=tool_call.id
                    ))
                continue  # 继续让LLM决定下一步
            
            # LLM输出最终结果
            return response.content
```

**规范要点**：
1. ✅ 使用`llm.bind_tools()`绑定工具（LangChain v1.0标准）
2. ✅ 所有提示词从配置文件加载，禁止硬编码
3. ✅ 代码只执行LLM决定的工具调用，不做业务判断
4. ✅ 设置`max_iterations`防止无限循环
5. ✅ 工具描述清晰，让LLM知道何时调用哪个工具

### 10.5 响应解析规范

**❌ 禁止复杂解析**：
```python

def parse_response(self, content):

    # 尝试解析Python dict
    # 尝试修复截断JSON
    # ...多层嵌套解析

**✅ 简单解析**：
```python
# 直接提取SQL
def _extract_sql(self, content):
    # 清理Markdown
    text = re.sub(r'^```.*?\n?', '', content)
    # 提取SQL语句
    match = re.search(r'(SELECT|INSERT|UPDATE|DELETE)\s+.+', text, re.I)
    return match.group(0) if match else None
```

1. **零硬编码**：所有提示词、业务逻辑从配置加载

6. **EditorConfig**：配置编辑器统一风格

**v6.0重构要点**：
- ✅ 使用LangChain v1.0 `bind_tools` API
- ✅ 移除所有硬编码提示词
- ✅ 移除所有硬编码业务逻辑
- ✅ LLM完全自主决策工具调用
- ✅ 提示词全部移至配置文件
